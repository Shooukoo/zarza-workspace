1. Introducción.
Propósito del sistema: Definir la arquitectura técnica y el diseño de datos para la plataforma Zarza AI, asegurando la trazabilidad de los muestreos en campo y la escalabilidad del procesamiento de imágenes.
Objetivos de diseño: Construir un sistema modular, tolerante a fallos de red en zonas rurales (offline-first para la app móvil) y centralizado en un único motor de inferencia de IA.
Definiciones y abreviaturas:
API RESTful: Interfaz de comunicación web (Application Programming Interface) que permite el intercambio de datos estructurados entre las aplicaciones cliente (Web/Móvil) y el servidor central mediante solicitudes HTTP.
Bcrypt: Algoritmo de encriptación unidireccional utilizado para proteger de forma segura las contraseñas de los usuarios en la base de datos.
Bucket / Object Storage: Servicio de almacenamiento en la nube optimizado para alojar archivos estáticos pesados y no estructurados (en este caso, las fotografías originales de las zarzamoras). Zarza AI utiliza Cloudflare R2 para este propósito.
Docker: Plataforma que permite empaquetar el software backend y sus dependencias en "contenedores", asegurando que el sistema se ejecute de manera idéntica y predecible en cualquier entorno de servidor.
FCM (Firebase Cloud Messaging): Servicio externo utilizado por el Backend para disparar notificaciones push y alertas a los dispositivos móviles de los Monitores y Agrónomos.
Fenología / Fenológico: Estudio de las etapas de desarrollo, crecimiento y maduración de los cultivos de zarzamora (ej. etapa de floración, fruto rojo, fruto maduro).
GeoJSON: Estándar de formato abierto diseñado para representar características geográficas (coordenadas de latitud y longitud). Se utiliza en MongoDB para trazar la ubicación exacta de los muestreos.
Human-in-the-loop: Modelo de interacción donde un experto humano (el Agrónomo) tiene la capacidad de revisar, validar o corregir los diagnósticos emitidos por la Inteligencia Artificial, garantizando la calidad de los datos.
JWT (JSON Web Token): Estándar de la industria utilizado para la creación de tokens de acceso seguro, permitiendo autenticar a los usuarios en el sistema sin necesidad de mantener sesiones activas en la memoria del servidor.
MongoDB: Base de datos NoSQL orientada a documentos que permite almacenar estructuras de datos flexibles y realizar consultas geoespaciales de alto rendimiento.
NestJS / Node.js: Framework y entorno de ejecución sobre los que está construido el servidor principal (Backend core) de la plataforma.
Offline-first: Enfoque de diseño de software donde la aplicación móvil se construye para operar de manera óptima y sin interrupciones sin conexión a internet, tratando la conectividad como una mejora secundaria y no como un requisito estricto.
RBAC (Role-Based Access Control): Sistema de seguridad arquitectónica que restringe el acceso a la información y a los endpoints del servidor dependiendo del rol asignado al usuario (Admin, Productor, Agrónomo, Monitor).
UUID (Universally Unique Identifier): Identificador alfanumérico único generado localmente por el dispositivo móvil para etiquetar de manera segura un muestreo temporal, evitando registros duplicados durante la sincronización.
Worker (AI Worker): Servidor o proceso secundario que se ejecuta de forma aislada para manejar tareas de alto consumo de recursos (correr el modelo de Machine Learning) sin bloquear o ralentizar el tráfico del servidor principal.
Referencias: Documento de Especificación de Requisitos de Software Versión 2.0.
2. Arquitectura del Sistema.
Esta sección describe la arquitectura técnica de Zarza AI, proporcionando una vista de alto nivel de los componentes, su separación de responsabilidades y las consideraciones operativas para garantizar la escalabilidad en el entorno agrícola.
2.1 Visión Global
El sistema Zarza AI opera bajo una arquitectura Cliente-Servidor basada en la nube. Está diseñado para centralizar el procesamiento pesado en un entorno controlado, mientras distribuye la recolección de datos a través de aplicaciones móviles preparadas para operar en entornos con nula o baja conectividad (Offline-first).
La arquitectura se divide en tres capas lógicas principales:
Capa de Presentación: Aplicación móvil para el trabajo de campo y plataforma web (Dashboard) para la administración y análisis.
Capa de Lógica de Negocio y API: Un servidor central que orquesta las peticiones, valida la jerarquía de roles, envía notificaciones y se comunica con el motor de IA.
Capa de Persistencia y Procesamiento: Base de datos estructurada para consultas, almacenamiento remoto de archivos estáticos (imágenes) y el servidor aislado (Worker) que ejecuta el modelo de Machine Learning.
2.2 Descomposición en Subsistemas
El ecosistema se desglosa en los siguientes subsistemas interconectados:
Subsistema Cliente Móvil (App): Responsable de interactuar con el hardware del dispositivo (Cámara, GPS). 
Contiene un módulo de almacenamiento local para encolar las fotografías e información de los muestreos cuando no hay internet.
Incluye un gestor de sincronización que escucha el estado de la red para enviar los datos al servidor en cuanto se detecta conexión.
Subsistema Cliente Web (Dashboard): 
Interfaz rica implementada para navegadores web.
Consume la API para renderizar gráficos dinámicos.
Subsistema Backend (API REST Core): 
Desarrollado en NestJS. Es el controlador de tráfico de Zarza AI.
Implementa la lógica de autenticación (JWT) y el Control de Acceso Basado en Roles (RBAC) para Admin, Productor, Agrónomo y Monitor.
Subsistema de Inteligencia Artificial (AI Worker): 
Un servicio aislado dedicado exclusivamente a recibir URLs de imágenes, correr el modelo de reconocimiento fenológico/patológico y devolver un JSON con las métricas de salud. No interactúa directamente con los usuarios ni con la app móvil.
Subsistema de Persistencia: 
Transaccional: Base de datos MongoDB para almacenar usuarios, campos, tareas operativas y los resultados de los muestreos (incluyendo índices GeoJSON).
Almacenamiento de Objetos (Object Storage): Un servicio bucket externo de Cloudflare R2 para alojar las imágenes originales capturadas en campo.
2.3 Interfaces
La comunicación entre los subsistemas se realiza mediante interfaces estandarizadas:
API RESTful: El Backend expone endpoints HTTP (GET, POST, PUT, DELETE) en formato JSON para la comunicación con la App Móvil y la Web.
Interfaz de Notificaciones: Integración con Firebase Cloud Messaging (FCM) para que el Backend pueda disparar "Solicitudes de Muestreo" hacia los dispositivos móviles de los trabajadores.
Interfaz de Almacenamiento (Cloud Storage API): Conexión segura desde el Backend hacia el bucket de imágenes (R2) mediante credenciales de acceso privadas, obteniendo URLs públicas o firmadas para que la IA pueda leerlas.
2.4 Casos de Uso Principales
A nivel arquitectónico, los flujos primarios que dictan el comportamiento del sistema son:
CU-01: Sincronización de Muestreos (Offline a Online): El Monitor recolecta datos sin red. Al conectarse, la App envía un lote de registros (UUID local, ubicación, IDs de campo). El Backend guarda las imágenes en el bucket, envía las URLs a la IA, recibe los diagnósticos, unifica la información en MongoDB y confirma a la App que puede borrar su caché local.
CU-02: Consulta de Gráficas de Producción: El Productor/Agrónomo solicita ver el rendimiento de un campo en la Web. El Backend valida sus permisos (campos_asignados), agrupa los datos históricos de los muestreos y devuelve las métricas consolidadas para renderizar las gráficas de comportamiento.
CU-03: Asignación de Tareas de Campo: El Admin crea una Solicitud de Muestreo en la Web. El Backend guarda el registro en MongoDB y dispara un evento a FCM, el cual despierta la App Móvil del Monitor asignado mediante una notificación push.
CU-04: Corrección de Diagnóstico (Human-in-the-loop): El Agrónomo visualiza un muestreo en la Web y detecta que la IA se equivocó. Envía una actualización; el Backend marca el registro con la bandera fue_corregido: true, guarda el diagnóstico original para auditoría y actualiza el diagnóstico final.
2.5 Aspectos de Seguridad, Rendimiento y Despliegue
Seguridad:
Autenticación sin estado mediante JSON Web Tokens (JWT).
Las contraseñas se almacenan con hashing unidireccional (bcrypt).
Autorización estricta por endpoint: El Backend verifica el rol del usuario extraído del JWT antes de ejecutar cualquier consulta.
Conexiones cifradas mediante HTTPS/TLS 1.2+ en todas las interfaces.
Rendimiento:
Desacoplamiento de archivos pesados: La base de datos no guarda imágenes, solo guarda URLs de texto ligero, evitando el colapso de MongoDB.
Uso de índices compuestos en MongoDB para garantizar que las búsquedas y filtrados por Productor y Campo respondan en milisegundos.
Procesamiento asíncrono en la IA para evitar que el hilo principal del Backend de Node.js se bloquee mientras se analizan las fotos.
Despliegue:
El sistema backend y el worker de IA se empaquetarán en contenedores (Docker) para garantizar que los entornos de desarrollo, pruebas y producción sean idénticos.
La infraestructura se alojará en proveedores de nube escalables, permitiendo aumentar los recursos de forma independiente según la demanda.
