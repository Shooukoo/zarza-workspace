1. Introducción
1.1 Propósito del Documento
Este Documento de Diseño de Software (SDD) define la arquitectura técnica, los patrones de diseño, la descomposición en subsistemas y los flujos de datos de la plataforma Zarza AI. Su objetivo es servir de guía de implementación para el equipo de desarrollo y como referencia de decisiones arquitectónicas para el equipo de ingeniería.
1.2 Objetivos de Diseño
Modularidad: Cada subsistema tiene responsabilidades claramente delimitadas y puede evolucionar de forma independiente.
Tolerancia a fallos de red: La app móvil opera en modo Offline-First, tratando la conectividad como una mejora y no como un requisito.
Centralización de la IA: Un único motor de inferencia en la nube garantiza resultados estandarizados para todos los canales (móvil y web).
Seguridad por diseño: Autenticación JWT y control de acceso RBAC aplicados en cada capa del sistema.
Escalabilidad horizontal: Arquitectura desacoplada mediante RabbitMQ que permite escalar el Worker de IA de forma independiente al backend.
1.3 Alcance
Este documento cubre el diseño de: la arquitectura de tres capas, la descomposición en subsistemas, los diagramas de flujo de los procesos críticos, el diseño de la base de datos (esquemas MongoDB), los patrones de diseño aplicados, las interfaces entre componentes y los principios de seguridad del sistema.

2. Arquitectura del Sistema
Esta sección describe la arquitectura técnica de Zarza AI, proporcionando una vista de alto nivel de los componentes, su separación de responsabilidades y las consideraciones operativas para garantizar la escalabilidad en el entorno agrícola.
2.1 Visión Global — Arquitectura de Tres Capas
Zarza AI opera bajo una arquitectura Cliente-Servidor basada en la nube, diseñada para centralizar el procesamiento pesado en un entorno controlado mientras distribuye la recolección de datos a través de aplicaciones móviles preparadas para operar en entornos con nula o baja conectividad (Offline-First).

2.3 Descomposición en Subsistemas
2.3.1 Subsistema Cliente Móvil (App Flutter)
Responsable de interactuar con el hardware del dispositivo (Cámara, GPS).
Módulo de almacenamiento local: encola fotografías y metadatos de muestreos cuando no hay internet, usando una base de datos local (SQLite o Hive).
Gestor de sincronización: escucha el estado de la red mediante un listener de conectividad. Al detectar conexión, envía los muestreos pendientes al backend con su offline_sync_id.
Módulo de notificaciones: recibe alertas de solicitudes de muestreo enviadas por FCM desde el backend.
Roles que lo usan: Monitor, Productor, Agrónomo.
2.3.2 Subsistema Cliente Web (Dashboard)
Interfaz de administración y consulta implementada para navegadores web.
Consume la API REST del backend para renderizar gráficos dinámicos de producción, mapas de calor y tablas de análisis.
Adapta las vistas según el rol del usuario autenticado (RBAC en frontend): el Admin ve todas las huertas; Productor y Agrónomo solo ven sus campos_asignados.
Roles que lo usan: Administrador, Productor, Agrónomo.
2.3.3 Subsistema Backend — API REST Core (fruit-backend)
Desarrollado en NestJS sobre Node.js. Es el controlador de tráfico principal de Zarza AI.
Gestiona la autenticación (JWT) y el Control de Acceso Basado en Roles (RBAC) mediante guards en cada endpoint.
Recibe imágenes del cliente, las sube a Cloudflare R2 y publica el evento nueva_fruta en RabbitMQ.
Gestiona el envío de notificaciones push a monitores a través de Firebase Cloud Messaging (FCM).
2.3.4 Subsistema Microservicio Consumidor (fruit-ms)
Consume los eventos de la cola RabbitMQ de forma asíncrona.
Al recibir un evento nueva_fruta, realiza una petición HTTP interna al Worker de IA (fruit-inference).
Mapea la respuesta JSON de la IA a las entidades de dominio y persiste el análisis en MongoDB.
El desacoplamiento mediante RabbitMQ garantiza que los picos de carga no afecten la disponibilidad del backend principal.
2.3.5 Subsistema de Inteligencia Artificial (fruit-inference)
Servicio aislado en Python con FastAPI, dedicado exclusivamente a la inferencia fenológica.
Descarga la imagen desde Cloudflare R2 usando boto3, ejecuta el modelo YOLOv8 y retorna un JSON estructurado con métricas de salud y cronograma fenológico.
No interactúa directamente con usuarios ni con la app móvil. Opera únicamente en la red interna privada de Docker.
Permite actualizar el modelo (best.pt) reiniciando únicamente este contenedor, sin afectar al resto del sistema.
2.3.6 Subsistema de Persistencia
MongoDB Atlas (Transaccional): Almacena usuarios, campos, análisis y solicitudes de muestreo. Utiliza índices 2dsphere para consultas geoespaciales y índices compuestos para filtrado por productor y campo.
Cloudflare R2 (Object Storage): Almacena las imágenes originales capturadas en campo. La base de datos solo guarda el storage_key (una ruta de texto liviana), evitando el colapso de MongoDB con archivos binarios pesados.

5.2 Decisiones de Diseño de la Base de Datos
Decisión
Justificación
Snapshot del requester en analyses
En lugar de una referencia directa, se guarda una copia del userId y email del solicitante. Esto garantiza trazabilidad histórica aunque el usuario sea eliminado del sistema.
offline_sync_id con índice Unique Sparse
Al ser Sparse, el índice solo aplica a documentos que contienen el campo, evitando conflictos de unicidad en muestreos web donde este campo no existe.
GeoJSON Point con índice 2dsphere
Permite consultas geoespaciales nativas de MongoDB: buscar todos los muestreos dentro de un radio, filtrar por polígono de una huerta, etc.
Imágenes en Cloudflare R2, no en MongoDB
Las imágenes pueden pesar varios MB. Almacenarlas en MongoDB degradaría el rendimiento de las consultas y dispararía los costos de almacenamiento transaccional.
Estado como Enum en solicitudes_muestreo
Usar una cadena controlada (PENDIENTE | EN_PROGRESO | COMPLETADO | CANCELADO) en lugar de booleanos facilita el filtrado y la legibilidad de los estados del flujo de trabajo.
6. Patrones de Diseño Aplicados
El diseño de Zarza AI aplica los siguientes patrones de software reconocidos para garantizar mantenibilidad, escalabilidad y separación de responsabilidades.
Patrón
Dónde se Aplica
Beneficio Obtenido
Event-Driven Architecture (EDA)
RabbitMQ entre fruit-backend y fruit-ms
Desacopla la ingesta de imágenes de la inferencia de IA. El backend no espera la respuesta del Worker, permitiendo respuestas HTTP rápidas al cliente.
Repository Pattern
Capa de acceso a datos en NestJS (Mongoose)
Separa la lógica de negocio de la lógica de persistencia. Facilita el reemplazo de MongoDB sin impactar los servicios.
Guard Pattern (RBAC)
JwtAuthGuard + RolesGuard en NestJS
Centraliza la lógica de autorización en decoradores reutilizables. Cada endpoint declara qué roles puede acceder sin duplicar código.
Idempotency Key
offline_sync_id en el flujo offline
Permite que la app reintente el envío de un muestreo sin crear duplicados, garantizando exactamente un registro por captura.
Snapshot Pattern
Campo requester en analyses
Preserva el estado del usuario en el momento del análisis. Garantiza trazabilidad histórica ante modificaciones o eliminaciones futuras.
Sidecar / Worker Isolation
fruit-inference como contenedor independiente
Aísla el proceso de alto consumo de CPU/GPU (YOLOv8) del servidor principal. Permite escalar el Worker de forma independiente según la carga de inferencia.
Offline-First / Local Cache
Módulo de sincronización en Flutter
Prioriza la experiencia del usuario en entornos sin red. Los datos se capturan siempre y se sincronizan oportunistamente al detectar conectividad.
7. Consideraciones de Seguridad, Rendimiento y Despliegue
7.1 Seguridad
Autenticación sin estado mediante JSON Web Tokens (JWT). Los tokens tienen tiempo de expiración configurable (JWT_EXPIRES_IN).
Contraseñas almacenadas con hashing unidireccional bcrypt (factor de costo mínimo: 12). Nunca se exponen en respuestas de la API.
Autorización estricta por endpoint: el Guard extrae el rol del JWT y verifica los permisos antes de ejecutar cualquier consulta o acción.
El Worker de IA (fruit-inference) opera exclusivamente en la red interna privada de Docker y nunca está expuesto a internet.
En producción, todas las comunicaciones externas deben cifrarse con TLS 1.2+ (HTTPS). Las URLs de imágenes en R2 deben ser URLs firmadas con tiempo de expiración.
7.2 Rendimiento
Desacoplamiento de archivos pesados: MongoDB solo almacena el storage_key (texto), nunca los binarios de las imágenes.
Índices compuestos en MongoDB (campo_id + productor_id, fecha_analisis) para garantizar que las búsquedas y filtrados respondan en milisegundos incluso con miles de registros.
Procesamiento asíncrono mediante RabbitMQ: el endpoint de ingesta retorna HTTP 201 inmediatamente al cliente sin esperar el resultado de la IA. La inferencia ocurre en segundo plano.
El Worker de IA puede instanciarse con múltiples réplicas en Docker Compose para distribuir la carga de inferencia en períodos de alta sincronización.
7.3 Despliegue
El sistema backend y el Worker de IA se empaquetan en contenedores Docker con imágenes base minimalistas (node:20-alpine y python:3.11-slim).
Docker Compose orquesta todos los servicios (fruit-backend, fruit-ms, fruit-inference, rabbitmq) garantizando entornos idénticos en desarrollo, pruebas y producción.
Las variables de entorno sensibles se gestionan mediante archivos .env y nunca se incluyen en el código fuente ni en las imágenes Docker.
La actualización del modelo YOLOv8 (best.pt) se realiza reiniciando únicamente el contenedor fruit-inference, sin interrumpir el backend ni la app.

CU-01
Sincronización Offline → Online
El Monitor recolecta datos sin red. Al conectarse, la App envía un lote de registros con UUID local, ubicación e IDs de campo. CU-03

8. Registro de Decisiones Arquitectónicas (ADR)
El siguiente registro documenta las decisiones de diseño más relevantes, su justificación y las alternativas consideradas.
ADR-01: MongoDB como Base de Datos Principal
Atributo
Detalle
Estado
Aceptado
Contexto
El sistema maneja datos de análisis con estructura variable (etapas fenológicas pueden variar) y requiere consultas geoespaciales de alto rendimiento.
Decisión
Usar MongoDB Atlas como base de datos principal.
Justificación
Soporte nativo para GeoJSON y el índice 2dsphere permiten consultas geoespaciales sin extensiones adicionales. El esquema flexible de documentos facilita la evolución del modelo de análisis sin migraciones costosas.
Alternativas Consideradas
PostgreSQL con extensión PostGIS (mayor rigidez de esquema), Firebase Firestore (limitaciones en consultas geoespaciales complejas).
ADR-02: RabbitMQ como Bus de Mensajes
Atributo
Detalle
Estado
Aceptado
Contexto
La inferencia de YOLOv8 puede tomar varios segundos. Bloquear el hilo principal de Node.js durante ese tiempo degradaría la experiencia del cliente.
Decisión
Usar RabbitMQ (AMQP) para desacoplar la ingesta de imágenes de la inferencia de IA.
Justificación
El backend responde al cliente inmediatamente (HTTP 201) y el Worker procesa la imagen en segundo plano. Los eventos se encolan si el Worker está ocupado, garantizando que ningún análisis se pierda.
Alternativas Consideradas
Llamada HTTP síncrona directa al Worker (bloquea el backend), BullMQ con Redis (más complejidad de configuración).
ADR-03: Flutter para la Aplicación Móvil
Atributo
Detalle
Estado
Aceptado
Contexto
Se requiere una app que funcione en Android con soporte offline, acceso a cámara y GPS, y capacidad de sincronización en segundo plano.
Decisión
Usar Flutter (Dart) como framework de desarrollo móvil.
Justificación
Base de código única para iOS y Android, rendimiento cercano a nativo gracias a la compilación AOT, amplio ecosistema de paquetes para SQLite (offline), GPS y FCM.
Alternativas Consideradas
React Native (mayor overhead de bridge), desarrollo nativo separado Android/iOS (doble costo de mantenimiento).
